Operation Path Planning (Suzanna Jiwani)

Given a known map, starting position, and end position, the goal of Operation Path Planning is to find the most optimal path from the start to the end that avoids obstacles. In our case, an optimal path is the shortest path, but you can also find other optimal paths for whatever you’re trying to minimize. We decided to approach the problem of finding the shortest path between start and end by transforming the problem into a graph problem, where each pose or pixel in the map was a graph node and edges were simply the straight lines connecting the nodes.

Once the map was created into an abstract, discrete graph using the pixels given in the map, we were able to generate a path by using a traditional search algorithm, “A*” where the weight or cost from traveling from one node to a neighboring node would be the distance in meters to get to another node from the connecting node. This algorithm works by starting at the start node and moving outwards, building paths one node at a time. It adds every path to a ‘queue’ sorted by overall distance where it would be picked later in the algorithm. Once the shortest current path from the start is picked from the queue, you look at the last node in the path and add all of its adjacent nodes to the end of the path, adding the newly created paths to the queue. The algorithm continues to do this until it reaches the goal, and in this way, it can be sure that once it reaches the goal, it has found the shortest path to get there.

The most unique aspect about the A* algorithm is that it also takes into account a ‘heuristic’ in order to guide the paths in the direction of the end node. It does this by weighting paths that end further from the goal more heavily than paths that end closer to the goal, even if each path is the same distance. This can drastically reduce the number of paths that the algorithm considers. It should be noted that since our heuristic is the direct distance from a node to the goal node: the cumulative sum of the distance it took to get to that node plus the heuristic of reaching the goal node is always going to be less than or equal to the total distance needed for a valid path to reach the goal node. This guarantees that a found path from a start node to goal node would be the shortest path possible. More formally, this approach upholds triangle equality and guarantees that the shortest possible path is found as a consequence.

Operation Pure Pursuit (Chrisitan)

Operation Pure Pursuit allows our autonomous vehicle to follow a given set of trajectories using our location estimation system (Particle Filter). The autonomous vehicle is aware of it’s beginning position and follows a set of trajectories as they are given. Operation Pure Pursuit uses this information to follow along a given trajectory. Until more information is given, the autonomous vehicle “drives” along this given trajectory until a new trajectory is sent. As it “drives”, the vehicle updates its estimated location using our location estimation system to be sent and used for more current, updated trajectories.

There are three main steps for pure pursuit. The first is to find the nearest point on the trajectory to the robot. The second is to find the point where a circle centered on the robot with a specified “lookahead” radius intersects with the trajectory ( further along the path than the closest point). Finally, the robot drives towards this “lookahead” point using Ackermann controls. The bullets below outline a more thorough yet high level explanation of the series of steps that go into pure pursuit:

  1. Listen to Trajectory Topic: The robot listens to a topic published path planner and creates an array of coordinate points that define a path.
  2. Update the Pose of the Robot: Once we receive a trajectory, we then initialize a subscriber which listens to and odometry topic and updates tha pose of the robot.
  3. Determine the Closest Point Each Line Segment: By parameterizing an equation for each line segment in the path, we find the closest point on each line segment to the current robot pose. From these distances, we find the closest line segment to the car. Thus, we can ignore any line segment the car has already passed by only using segments with indices larger than the closest segment.
  4. Find the Goal Point: We find the intersection of a circle with radius equal to the lookahead distance and the line segments on the path to find a goal point to use as a steering heading.
  5. Calculate the Steering Angle: The lookahead goal point is used to determine the angle used in Ackerman steering control, and the car drives towards the goal point until new trajectory segments are used.

Integration

Localization code determines the starting point of the robot, while a click in the simulation determines a goal position for the robot. Operation Path Planning creates a trajectory for the autonomous vehicle to execute in order to reach this goal position on the map. Operation Path Planning feeds this segmented trajectory into Operation Pure Pursuit and then the control code within Pure Pursuit steers the robot (as outlined above) along the pre-generated path. By integrating the localization, planning, and pursuit components we have already created, we will be able to click anywhere in simulation, generate a path that avoids obstacles, and watch as the robot drives autonomously and uninhibited to that location.
